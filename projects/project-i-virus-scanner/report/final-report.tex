\begin{abstract}
Sentinel is a safe, read-only, Rust-based educational virus scanner for Project
I in Network Security Practice. The scanner accepts an explicit target path,
loads a structured JSON signature database, computes MD5 and SHA-256 digests
incrementally, uses a Bloom filter as a negative pre-check before exact hash-map
verification, searches byte signatures with a streamed Aho-Corasick automaton,
skips symbolic links by default, separates heuristic warnings from confirmed
signature matches, and writes JSON plus Markdown reports. The heuristic layer
implements three bounded static checks: process-injection API names,
executable-like magic bytes inside text-like files, and high byte entropy. This
report is the canonical Project I report for the Rust submission path. It adds
the Rust execution template and documents the algorithm and implementation
excerpts in an academic-paper style. The demo uses the official EICAR safe
anti-malware test file generated by Rust at demo time, plus benign local
fixtures. No live malware is used.
\end{abstract}

\section{Problem Statement}

The official Project I task is to construct a virus scanner that can identify a
known threat inside a directory, compare files against a signature database, and
produce a security report. The central engineering problem is not simply to
print a finding; it is to build a reproducible, explainable pipeline where each
detected result can be traced back to the signature database, the matching
method, and the scan metadata.

The Rust version of Sentinel treats the project as a static-analysis scanner:
files are opened as bytes, read in bounded chunks, compared against configured
hash and byte-pattern signatures, and summarized without mutating the target
tree. Confirmed signature evidence is reported as \texttt{infected}. Weak
indicator-only evidence is reported as \texttt{suspicious}.

\section{Rust Heuristic Analysis}

The official brief asks for a basic heuristic ruleset to flag suspicious file
behavior. Sentinel keeps this layer deliberately bounded and non-destructive:
heuristics are computed from the same in-memory byte sample already read during
static scanning. A heuristic finding cannot override the signature result model;
without an exact hash or byte-pattern signature, it can only produce
\texttt{suspicious}.

\begin{longtable}{p{0.28\linewidth}p{0.32\linewidth}p{0.30\linewidth}}
\toprule
\textbf{Rule} & \textbf{Rust condition} & \textbf{Reason} \\
\midrule
\texttt{api-name-indicator} & The bounded text view contains process-injection
API names such as \texttt{CreateRemoteThread}, \texttt{VirtualAllocEx}, or
\texttt{WriteProcessMemory}. & Course-aligned example of suspicious behavior
without claiming confirmed malware. \\
\texttt{magic-mismatch} & A text-like file begins with executable magic bytes
such as \texttt{MZ} or \texttt{ELF}. & Flags a common hiding pattern: content
type does not match the visible extension. \\
\texttt{high-entropy-sample} & A sample of at least 256 bytes reaches
\texttt{7.5} bits per byte or higher. & Flags packed, encrypted, or compressed
content as weak evidence for review. \\
\bottomrule
\end{longtable}

The Rust report records this layer as
\texttt{heuristic\_engine=bounded-static-byte-sample} and lists the active rule
identifiers in \texttt{heuristic\_rules}. This makes the generated report
auditable: a reviewer can see not only the result, but which static rule family
was enabled during the scan.

\section{Core Algorithm}

\subsection{Algorithm 1: Rust Scan Pipeline}

\begin{lstlisting}[style=sentinel]
Input:
  target_path T
  signature_database_path S
  json_report_path J
  markdown_report_path M

Output:
  JSON report J
  Markdown report M
  process exit code 1 if confirmed infections exist, else 0

1. D <- load_signature_database(S)
2. P <- PatternEngine::new(D.patterns)
3. H <- D.hash_filter
4. R <- empty result list
5. For each path p collected from T in deterministic sorted order:
     a. If p is a symbolic link:
          append skipped result to R
          continue
     b. Open p as bytes
     c. Initialize MD5 context, SHA-256 context, and P.stream()
     d. While a chunk is available:
          update MD5 with chunk
          update SHA-256 with chunk
          feed chunk into the Aho-Corasick stream
          append chunk bytes to bounded heuristic sample if sample is not full
     e. For each computed digest d:
          if H.might_contain(algorithm:d) is false:
              skip exact lookup for that digest
          else:
              verify against D.hash_index[algorithm][d]
     f. Add byte-pattern matches returned by the stream scanner
     g. Run heuristic rules over the bounded sample
    h. Classify:
          if any exact signature or byte-pattern match -> infected
          else if any heuristic finding -> suspicious
          else -> clean
     i. append FileResult to R
6. Build summary and findings from R
7. Write J and M
8. Return exit code 1 if summary.infected > 0, else 0
\end{lstlisting}

\subsection{Algorithm 2: Streamed Aho-Corasick Matching}

\begin{lstlisting}[style=sentinel]
Build phase:
1. Insert every configured byte pattern into a trie.
2. For each terminal trie node, store the corresponding pattern index.
3. Traverse the trie breadth-first.
4. For each node and transition byte:
     a. Follow failure links until a fallback transition exists or root is hit.
     b. Store the fallback state as the child node's failure link.
     c. Inherit output pattern indexes from the failure state.

Scan phase:
1. Start with state = root and absolute offset = 0.
2. For each byte b in each file chunk:
     a. Follow failure links until state has transition b or root is reached.
     b. Move to the next state for b, or root if absent.
     c. For each output pattern at the current state:
          pattern_offset = absolute_position + 1 - pattern_length
          record signature id, matcher type, pattern length, and offset
3. Preserve state between chunks so split signatures still match.
\end{lstlisting}

\subsection{CLI Template and Exit Policy}

fn run(args: Vec<String>) -> Result<i32, String> {
    match args.first().map(String::as_str) {
        Some("--version") | Some("-V") => {
            println!("sentinel-v2-rust {VERSION}");
            Ok(0)
        }
        Some("validate-signatures") => {
            let signatures = parse_value_arg(&args[1..], "--signatures")
                .map(PathBuf::from)
                .unwrap_or_else(default_signature_path);
            let database = load_signature_database(&signatures)?;
            println!(
                "signature database ok: schema={} signatures={} hash_matchers={} patterns={}",
                database.schema_version,
                database.signatures.len(),
                database.hash_matcher_count(),
                database.patterns.len()
            );
            Ok(0)
        }
        Some("scan") => {
            let scan_args = parse_scan_args(&args[1..])?;
            let database = load_signature_database(&scan_args.signatures)?;
            let report = scan_directory(&scan_args.target, &database)?;
            write_json_report(&report, &scan_args.json_report)?;
            write_markdown_report(&report, &scan_args.markdown_report)?;

            if report.summary.infected > 0 {
                Ok(1)
            } else {
                Ok(0)
            }
        }
        _ => {
            print_usage();
            Ok(2)
        }
    }
}
\end{lstlisting}

\subsection{Chunked File Scan}

\begin{lstlisting}[style=sentinel]
fn scan_file_content(
    path: &Path,
    pattern_engine: &PatternEngine,
) -> Result<ScannedFileContent, String> {
    let mut file = File::open(path).map_err(|err| err.to_string())?;
    let mut md5_context = md5::Context::new();
    let mut sha256 = Sha256::new();
    let mut pattern_stream = pattern_engine.stream();
    let mut heuristic_sample = Vec::new();
    let mut size_bytes = 0_u64;
    let mut buffer = vec![0_u8; CHUNK_SIZE_BYTES];

    loop {
        let bytes_read = file.read(&mut buffer).map_err(|err| err.to_string())?;
        if bytes_read == 0 {
            break;
        }

        let chunk = &buffer[..bytes_read];
        md5_context.consume(chunk);
        sha256.update(chunk);
        pattern_stream.feed(chunk);
        size_bytes += bytes_read as u64;

        if heuristic_sample.len() < HEURISTIC_SAMPLE_LIMIT_BYTES {
            let remaining = HEURISTIC_SAMPLE_LIMIT_BYTES - heuristic_sample.len();
            heuristic_sample.extend_from_slice(&chunk[..chunk.len().min(remaining)]);
        }
    }

    Ok(ScannedFileContent {
        size_bytes,
        md5: format!("{:x}", md5_context.compute()),
        sha256: format!("{:x}", sha256.finalize()),
        pattern_matches: pattern_stream.finish(),
        heuristic_sample,
    })
}
\end{lstlisting}

\subsection{Bloom-Then-Exact Hash Verification}

\begin{lstlisting}[style=sentinel]
fn find_hash_matches(
    hashes: &HashMap<String, String>,
    database: &SignatureDatabase,
) -> Vec<HashSignatureMatcher> {
    let mut matches = Vec::new();
    for (algorithm, digest) in hashes {
        if !database.hash_filter.might_contain(algorithm, digest) {
            continue;
        }
        if let Some(digest_map) = database.hash_index.get(algorithm) {
            if let Some(found) = digest_map.get(digest) {
                matches.extend(found.iter().cloned());
            }
        }
    }
    matches
}
\end{lstlisting}

\subsection{Streamed Pattern Matching}

\begin{lstlisting}[style=sentinel]
pub fn feed(&mut self, data: &[u8]) {
    for (chunk_position, byte) in data.iter().copied().enumerate() {
        self.state = self.engine.transition(self.state, byte);
        let absolute_position = self.offset + chunk_position;

        for pattern_index in self.engine.nodes[self.state].outputs.iter().copied() {
            if self.seen.insert(pattern_index) {
                let pattern = &self.engine.patterns[pattern_index];
                let offset = absolute_position + 1 - pattern.pattern.len();
                self.matches.push(PatternMatch {
                    signature: pattern.signature.clone(),
                    matcher: pattern.matcher_type.clone(),
                    pattern_offset: offset,
                    pattern_bytes: pattern.pattern.len(),
                });
            }
        }
    }

    self.offset += data.len();
}
\end{lstlisting}

\subsection{Bounded Heuristic Rules}

\begin{lstlisting}[style=sentinel]
fn run_heuristics(path: &Path, content: &[u8]) -> Vec<HeuristicFinding> {
    let sample_len = content.len().min(1024 * 1024);
    let text = String::from_utf8_lossy(&content[..sample_len]);
    let mut findings = Vec::new();

    let indicators: Vec<&str> = ["CreateRemoteThread", "VirtualAllocEx", "WriteProcessMemory"]
        .into_iter()
        .filter(|indicator| text.contains(indicator))
        .collect();

    if !indicators.is_empty() {
        findings.push(HeuristicFinding {
            rule_id: "api-name-indicator".to_string(),
            severity: "medium".to_string(),
            description: "Suspicious API-name strings appear in the file.".to_string(),
            evidence: indicators.join(", "),
        });
    }

    if starts_with_executable_magic(content) && has_text_like_extension(path) {
        findings.push(HeuristicFinding {
            rule_id: "magic-mismatch".to_string(),
            severity: "high".to_string(),
            description: "Executable-like magic bytes appear in a text-like file.".to_string(),
            evidence: format!(
                "extension={} magic=MZ",
                path.extension()
                    .and_then(|value| value.to_str())
                    .unwrap_or("<none>")
            ),
        });
    }

    if content.len() >= HIGH_ENTROPY_MIN_BYTES {
        let entropy = shannon_entropy(content);
        if entropy >= HIGH_ENTROPY_THRESHOLD {
            findings.push(HeuristicFinding {
                rule_id: "high-entropy-sample".to_string(),
                severity: "low".to_string(),
                description: "The bounded sample has high byte entropy.".to_string(),
                evidence: format!("entropy={entropy:.2} bits/byte sample_bytes={}", content.len()),
            });
        }
    }

    findings
}
\end{lstlisting}

\section{Result Classification}

\begin{longtable}{p{0.18\linewidth}p{0.72\linewidth}}
\toprule
\textbf{Status} & \textbf{Rust condition} \\
\midrule
\texttt{infected} & At least one exact MD5/SHA-256 signature match or one
configured byte-pattern match is found. \\
\texttt{suspicious} & No signature match exists, but a heuristic indicator is
present in the bounded sample. \\
\texttt{clean} & No signature match and no heuristic finding exists. \\
\texttt{skipped} & The path is a symbolic link and the default symlink policy is
\texttt{skip}. \\
\texttt{error} & The scanner cannot read metadata or file content for the path. \\
\bottomrule
\end{longtable}

This ordering prevents heuristic-only evidence from being overstated as a
confirmed infection. It also makes the expected course demo easy to audit:
the generated EICAR safe test file is confirmed by MD5, SHA-256, and
hex-pattern evidence, while the API-name fixture remains only suspicious.

\section{Security Boundary and Ethics}

The scanner is intentionally conservative. The current demo uses the official
EICAR safe anti-malware test file because the project brief asks for EICAR, but
the literal \texttt{eicar.com.txt} file is generated at demo time by Rust and is
not committed or exported. EICAR is not live malware. The scanner does not
execute scanned files, detonate samples, mutate targets, quarantine files, or
upload telemetry. MD5 is used only as a legacy signature-matching demonstration
and is paired with SHA-256 and byte-pattern evidence; it is not presented as a
modern cryptographic integrity guarantee.

\section{Limitations}

Sentinel is not production antivirus software. Its coverage is limited to the
configured signatures and simple heuristic indicators. It does not implement
archive unpacking, PE parsing, sandbox execution, cloud reputation, YARA rules,
or real-time file-system monitoring. These omissions are intentional for the
course setting: the goal is a clear, reproducible, safe scanner whose algorithm
and outputs can be reviewed directly.

\section{Conclusion}

The Rust Sentinel implementation satisfies the Project I educational goal:
it scans an explicit directory, validates a structured signature database,
combines Bloom-filter pre-checking with exact hash verification, performs
stream-safe Aho-Corasick byte-pattern detection, reports weak heuristics
separately from confirmed signatures, and generates reproducible JSON,
Markdown, and evidence-manifest artifacts. This final Rust report documents both
the algorithm and the actual Rust template used to build, test, validate, and
run the scanner.
