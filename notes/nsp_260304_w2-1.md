# 📖 課程講義：環境變數與安全 (Environment Variables & Attacks)

## 一、 核心概念定義：什麼是環境變數？

**環境變數 (Environment Variables)** 是一組具名的動態值 (Dynamic named values)，存在於行程 (Process) 運作的作業環境中 。它們會直接影響執行中行程的行為模式 。
* **起源與應用**：最初在 Unix 系統中引入，後來微軟 Windows 系統也採用了此機制 。
* **經典範例**：`PATH` 變數。當我們在終端機輸入一個指令時，系統會依據 `PATH` 中定義的目錄順序去尋找對應的執行檔 。



### 1. 在 C 程式中存取環境變數

程式開發者主要有兩種方式可以在 C 語言中讀取環境變數 ：

1. **透過 `main` 函式參數**：`void main(int argc, char *argv[], char *envp[])`，其中 `envp` 即為環境變數陣列 。
2. **透過全域變數 `environ` (推薦)**：宣告 `extern char **environ;` 。
* **💡 教授重點補充**：初始狀態下，`envp` 和 `environ` 指向記憶體中的同一個位置（Stack 區段）。但如果在程式執行期間新增了環境變數，系統可能會將這些變數移至 Heap 區段，此時全域變數 `environ` 會更新指向新位置，但 `main` 函式內的 `envp` 指標並不會改變，這可能會導致讀取錯誤 。


## 二、 行程繼承與 Shell 變數的差異

### 1. 環境變數的傳遞方式

行程可以透過兩種主要系統呼叫 (System Call) 獲取或傳遞環境變數：

* **`fork()`**：當建立子行程時，子行程會完全繼承父行程的環境變數 。
* **`execve()`**：用於在當前行程中執行新程式。預設情況下，記憶體空間會被覆寫，舊的環境變數會消失 。但開發者可以透過 `execve` 的第三個參數 `envp[]` 來明確指定要傳遞給新程式的環境變數陣列 。



### 2. Shell 變數 vs. 環境變數

這兩者經常被混淆，但它們在系統中的運作機制不同 。

* **Shell 變數 (Shell Variables)**：是 Shell 程式（如 bash）內部使用的變數。你可以透過 `FOO=bar` 來建立，它**不會**被傳遞給 Shell 啟動的子行程 。
* **環境變數**：當 Shell 啟動時，它會將系統環境變數複製成內部的 Shell 變數。如果使用者修改了 Shell 變數，並不會影響原本的環境變數 。必須透過 `export` 指令（例如 `export FOO=bar`），將 Shell 變數提升為環境變數，這樣後續啟動的子行程才能繼承到它 。



> **🛠️ 實務檢測技巧**：
> 教授在課堂中展示了如何使用虛擬檔案系統 `/proc` 來觀察行程真實的環境變數。你可以使用指令 `strings /proc/$$/environ` 來印出當前 Shell 行程的環境變數（`$$` 代表當前 Shell 的 PID）。

## 三、 攻擊面分析與實際案例 (Attack Surfaces)

由於一般使用者可以自由設定環境變數，如果「特權程式 (Set-UID programs)」在底層隱含使用了這些變數，環境變數就會成為一個危險的**攻擊面 (Attack Surface)** 。

### 攻擊面一：動態連結器 (Dynamic Linker)

* **原理**：動態連結會在程式執行時才去尋找並載入外部函式庫 (`.so` 檔) 。這意味著部分程式碼在編譯時是未決定的 。如果使用者能控制連結器尋找函式庫的路徑，就能植入惡意程式碼 。
* **攻擊手法 (LD_PRELOAD)**：`LD_PRELOAD` 環境變數允許使用者指定「優先載入」的共享函式庫 。
* **案例**：假設程式呼叫了標準庫的 `sleep()`。攻擊者可以自己寫一個惡意的 `sleep()` 函式，編譯成共享函式庫 `libmylib.so`，然後設定 `export LD_PRELOAD=./libmylib.so`。當程式執行時，就會觸發攻擊者寫的惡意函式，而不是系統原生的 `sleep()` 。
* **系統防禦機制**：對於 Set-UID 程式（例如具有 root 權限的程式），動態連結器會檢查 Effective UID (EUID) 和 Real UID (RUID) 是否一致。如果不一致，連結器會主動忽略 `LD_PRELOAD` 和 `LD_LIBRARY_PATH` 等危險變數，以防止權限提升攻擊 。

這是一份為您整理的**「作業系統與系統安全：動態連結器與權限防禦機制」**進階課程講義。本講義針對 Set-UID 程式的底層防禦邏輯進行了深度拆解，這對於理解系統漏洞利用（Exploitation）與防禦策略至關重要。

#### Dynamic Linker 核心觀念解析

在 Linux/Unix 作業系統中，環境變數（Environment Variables）是影響程式執行行為的重要參數。然而，當使用者執行具有高權限的 Set-UID 程式時，若允許惡意使用者隨意操控如 `LD_PRELOAD` 這類會影響「動態連結器（Dynamic Linker）」行為的變數，將引發嚴重的資安危機。

為此，作業系統在動態連結器中實作了嚴格的**身分核對機制**：透過比對程式執行時的**真實身分 (RUID)** 與**有效身分 (EUID)**，來決定是否過濾掉危險的環境變數，從根本上阻斷權限提升的攻擊路徑。

---

#### 專有名詞定義與舉例

##### 1. Set-UID 程式 (Set User ID)

* **定義**：一種特殊的 Unix/Linux 檔案權限設定。當一般使用者執行帶有 Set-UID 屬性的執行檔時，該行程（Process）會「暫時」獲得該檔案擁有者的權限（通常是 root 權限），以便執行需要特權的操作。
* **舉例**：Linux 系統中的 `passwd` 指令。一般使用者需要透過它來修改自己的密碼，而修改密碼必須寫入受嚴格保護的 `/etc/shadow` 檔案。因此 `passwd` 被設定為 Set-UID root 程式。

##### 2. Real UID (RUID, 真實使用者 ID)

* **定義**：代表「究竟是誰啟動了這個行程」。它記錄了最初登入系統並下達指令的使用者身分。
* **舉例**：如果您的登入帳號是 `jason`（假設 UID 為 1000），當您在終端機輸入任何指令時，該行程的 RUID 就是 1000。

##### 3. Effective UID (EUID, 有效使用者 ID)

* **定義**：作業系統在執行期間，實際用來**檢查檔案存取與特權操作權限**的身分。
* **舉例**：當 RUID 為 1000 的 `jason` 執行了 Set-UID 為 root 的 `passwd` 程式時，雖然 RUID 還是 1000，但 EUID 會瞬間變成 0（root 的 UID）。作業系統此時會視該行程為 root，允許其修改密碼檔。

##### 4. 動態連結器 (Dynamic Linker)

* **定義**：在程式啟動時，負責將程式所需的外部共用函式庫（Shared Libraries，如 `.so` 檔）載入記憶體中並進行連結的系統程式（例如 `/lib/ld-linux.so`）。

##### 5. LD_PRELOAD 與 LD_LIBRARY_PATH (危險的環境變數)

* **定義**：
    * `LD_PRELOAD`：指示動態連結器在載入所有標準函式庫**之前**，強制優先載入指定的函式庫。
    * `LD_LIBRARY_PATH`：指示動態連結器去自訂的非標準目錄下尋找函式庫。
* **舉例**：攻擊者可以撰寫一個惡意的 `sleep()` 函式，並設定 `export LD_PRELOAD=./malicious.so`。當程式嘗試呼叫系統標準的 `sleep()` 時，就會被「劫持」，轉而執行攻擊者的惡意程式碼。

#### 現實生活中的應用實例：特勤人員的安全查核

我們可以將這個防禦機制想像成一個**高機密金庫的門禁管制系統**：

* **RUID (真實身分)**：您是一位基層警員（權限低）。
* **EUID (有效身分)**：您今天被指派拿著「局長授權的特別通行證」（Set-UID 程式），需要進入證物室（執行特權操作）。
* **LD_PRELOAD (危險變數/自備工具)**：您因為私人習慣，想帶自己外面買的非制式開鎖工具（自訂函式庫）進去證物室使用。
* **動態連結器 (安檢警衛)**：
    * 安檢警衛在門口攔下您進行身分核對。他發現您的真實身分（RUID=基層警員）與您手上拿的特別通行證（EUID=局長）**「不一致 (EUID != RUID)」**。
    * 警衛心想：「雖然你有局長的特別授權可以進去，但既然你不是局長本人，我就絕對不能讓你帶外面的不明工具（主動忽略 `LD_PRELOAD`）進去操作，你只能用證物室裡標準配置的官方工具（標準 C 函式庫）。」

透過這道防線，系統成功防止了基層人員利用高階通行證，夾帶惡意工具進行內部破壞（防止權限提升攻擊）。

#### 延伸教材補充：滲透測試與系統底層防禦

在進行系統滲透測試（Penetration Testing）或是分析底層資安架構時，這段防禦邏輯是攻防雙方的必爭之地：

##### 1. 安全邊界的極限：如果 EUID 與 RUID 相符呢？

如果動態連結器發現 `EUID == RUID`（例如您原本就是 root 帳號，或者您執行的是一般的非 Set-UID 程式），防禦機制就會**解除限制**，允許載入 `LD_PRELOAD`。
在軟體開發與除錯上，這是一項極佳的特性，開發者可以利用 `LD_PRELOAD` 輕易地攔截並測試系統呼叫（Hooking System Calls）；但一旦作業系統的核心元件設定錯誤（例如錯誤地將某個程式的權限設定與擁有者對齊），這就會立刻成為駭客提權的跳板。

##### 2. 應用程式層級的環境變數防禦：`secure_getenv()`

動態連結器的保護只涵蓋了「函式庫載入階段」。如果 Set-UID 程式本身在程式碼內部使用了 `getenv("PWD")` 或 `getenv("PATH")` 等標準 API 來讀取環境變數，依然會面臨緩衝區溢位或指令注入的風險。
因此，在撰寫高強度的 C/C++ 特權程式時，資安規範強烈建議改用 GNU 延伸的 **`secure_getenv()`** 函式。其底層實作邏輯與動態連結器如出一轍：當它偵測到 `EUID != RUID` 時，會直接回傳 `NULL`，強制程式以最安全的預設狀態運行，拒絕一切來自使用者的不可信環境變數輸入。

##### 3. 架構層級的演進：為何現代系統逐漸淘汰 Set-UID？

正因為 Set-UID 程式面臨著環境變數（Environment Variables）所帶來的巨大且複雜的攻擊面，現代作業系統（例如 Android）已經在架構上**全面移除了 Set-UID 機制**。取而代之的是採用**「服務導向途徑 (Service Approach)」**：將需要特權的操作隔離成獨立的背景守護行程（Daemon），一般使用者只能透過嚴格定義的進程間通訊（IPC）發送請求，由守護行程代為執行，從物理層面上徹底切斷了環境變數感染特權程式的可能。

### 攻擊面二：外部程式呼叫 (External Programs)

* **`system()` 的危險性**：開發者常使用 `system("cal")` 來呼叫外部程式。`system()` 底層其實是呼叫 `/bin/sh -c "cal"` 。
* **攻擊手法 (PATH 注入)**：因為沒有寫出絕對路徑（如 `/usr/bin/cal`），Shell 會依賴使用者的 `PATH` 環境變數來尋找 `cal` 執行檔 。攻擊者可以寫一個惡意腳本命名為 `cal`，並將自己所在的目錄加到 `PATH` 的最前面 (`export PATH=.:$PATH`)。當特權程式呼叫 `system("cal")` 時，就會以 root 權限執行攻擊者的腳本 。
* **防禦建議**：在特權程式中呼叫外部程式時，**絕對不要使用 `system()**`，應該使用 `execve()`，因為它不透過 Shell 執行，不受環境變數影響 。



### 攻擊面三：應用程式碼本身漏洞 (Application Code)

* **案例**：程式使用 `getenv("PWD")` 取得當前路徑，並直接 `sprintf` 到一個固定大小的陣列 `arr[64]` 中 。
* **攻擊手法**：攻擊者可以故意將環境變數 `PWD` 設定為一個超級長的字串，當程式讀取並複製時，就會引發**緩衝區溢位 (Buffer Overflow)** 漏洞 。
* **防禦建議**：應該使用安全的版本如 `secure_getenv()`。當系統偵測到需要安全執行（例如 EUID 不等於 RUID 的 Set-UID 程式）時，`secure_getenv()` 會回傳 NULL，避免讀取到惡意輸入 。

## 四、 現實生活中的應用實例與延伸思考

這部分結合了課程內容與您在資安領域的實務背景（包含滲透測試與系統管理）：

**1. CI/CD 流程中的機敏資料外洩 (DevOps 安全)**

* **實例**：在現代軟體開發中，我們經常使用環境變數來傳遞資料庫密碼、API 金鑰給 Docker 容器或 GitHub Actions 腳本。如果開發者在程式異常崩潰時，不慎將當下所有的環境變數（包含機敏資料）一併 dump 到 log 系統中，就會造成嚴重的資安事件。這呼應了課程中提到「環境變數也是一種 Input Data，必須被淨化 (Sanitize)」的概念 。



**2. 延伸教材：Rust 語言如何防範環境變數攻擊？**

* **補充**：鑑於您對開發高效能 AI 應用的 Rust 語言感興趣，Rust 在設計上預設就比 C/C++ 安全。在讀取環境變數時，Rust 使用 `std::env::var("KEY")`，它會回傳一個 `Result` 型別。如果變數不存在或包含無效的 Unicode 字元，程式必須顯式地處理這些錯誤（Pattern Matching），從根本上避免了 C 語言中常見的 `NULL` 指標解讀錯誤或字串長度溢位問題。此外，Rust 的 `std::process::Command` 預設行為類似於安全的 `execve`，除非開發者刻意呼叫 `Command::new("sh").arg("-c")`，否則不會受到 `$PATH` 注入攻擊的影響。

**3. 滲透測試 (CEH) 中的特權提升 (Privilege Escalation)**

* **實例**：在準備 CEH 認證的實務演練中，當你取得了一台 Linux 伺服器的低權限 shell 後，第一件事通常是尋找系統中有沒有配置不當的 Set-UID 程式。如果你發現某個 Set-UID 程式使用了 `system()` 呼叫外部指令，你就可以利用講義中提到的「修改 `$PATH` 變數」手法，輕鬆取得 Root 權限（如講義 Slide 27 所示 ）。



**4. 架構安全：為什麼 Android 放棄了 Set-UID？**

* **實例**：講義最後提到，由於 Set-UID 架構的攻擊面太廣（環境變數難以完全信任），現代作業系統如 Android 已經全面移除 Set-UID 機制 。取而代之的是「服務導向架構 (Service Approach)」，一般 App 如果需要特權操作（例如存取相機、GPS），必須發送請求 (Request for service) 給具有特權的系統 Daemon 處理。這種設計大大縮小了攻擊面，因為環境變數的影響力被隔離了 。
