# 課程講義：Host Security — Linux Security Basics 與 Set-UID Privileged Programs

## 一、課程定位與核心問題

在主機安全裡，最重要的不是「怎麼把所有人都擋掉」，而是：

1. **誰可以使用哪些資源**
2. **誰可以暫時取得更高權限**
3. **系統如何在提供便利的同時避免權限濫用**
4. **一旦程式有漏洞，是否會直接變成 root 級災難**

Linux/UNIX 的安全模型，本質上是在回答這幾件事：

* 使用者與群組怎麼分
* 檔案與目錄怎麼保護
* 一般使用者怎麼透過受控方式完成特權操作
* 為什麼 `/etc/shadow` 不能隨便讀，卻又必須讓使用者改密碼
* 為什麼 Set-UID 既必要又危險

# 二、先備核心觀念

## 1. Operating System（作業系統）

**定義**
作業系統是管理硬體資源、提供程式執行環境、並協調使用者與應用程式的核心軟體。書中將電腦系統拆成硬體、作業系統、應用程式與使用者四層。

**例子**
Linux、Windows、macOS、Android、iOS。

**現實應用**
你打開瀏覽器、存取檔案、連網、切換程式，背後都要靠 OS 進行 CPU、記憶體、I/O、權限管理。

---

## 2. Process（行程）

**定義**
程式載入記憶體後並開始執行，就成為 process。它是 OS 分配資源與權限的基本執行單位。OS Concepts 將 process 視為系統中的基本工作單位。 

**例子**
`passwd`、`sudo`、`bash`、`firefox` 都是 process。

**現實應用**
你登入 Linux 後執行 `ping`、`cat`、`passwd`，實際上都是不同 process 在請求系統資源。

---

## 3. User mode / Kernel mode（使用者態 / 核心態）

**定義**
現代作業系統至少有兩種執行模式：

* **User mode**：一般應用程式執行
* **Kernel mode**：OS 核心與特權操作執行

某些指令只能在 kernel mode 執行，稱為 **privileged instructions**。OS Concepts 指出像 I/O control、timer management、interrupt management 都屬於特權操作。 

**例子**
一般程式不能直接控制磁碟或網卡，必須透過 system call 進入 kernel 代辦。

**現實應用**
這是整個主機安全的根基。若沒有 user/kernel mode 分離，任何惡意程式都能直接控制硬體與系統。

---

## 4. System Call（系統呼叫）

**定義**
使用者程式向作業系統請求服務的正式介面。程式透過 system call 從 user mode 轉到 kernel mode，讓 OS 幫它做受保護的事。 

**例子**
`open()`、`read()`、`write()`、`fork()`。

**現實應用**
即使你只是開檔、讀檔、建立 socket，背後都是 system call。

---

# 三、Linux Security Basics

## 1. User（使用者）

**定義**
Linux 中每個使用者都有唯一的 UID。系統藉由 UID 追蹤「誰在做事」。投影片指出每位使用者都會有唯一 user ID。

**例子**

* `root`：超級使用者
* `alice`：一般使用者
* `www-data`：Web 服務帳號

**現實應用**

* Web server 不該用 root 執行
* DB service 通常會用專屬帳號
* CI/CD agent 也常用獨立 service account

---

## 2. Group（群組）

**定義**
群組是多個使用者的集合，用來簡化權限管理。使用者可以同時屬於多個群組。

**例子**

* `sudo` 群組：可透過 sudo 執行特權指令
* `docker` 群組：可操作 Docker
* `wireshark` 群組：可使用封包擷取能力

**現實應用**

* 實驗室伺服器上，把助教放同一群組方便管理共享檔案
* DevOps 團隊共用 deploy 權限
* 資安團隊共用 log 讀取權限

---

## 3. File permission（檔案權限）

**定義**
傳統 Linux 權限模型以三種身分管理：

* owner
* group
* others

每種身分可有：

* `r` read
* `w` write
* `x` execute

投影片明確整理了檔案與目錄的 `r/w/x` 意義。

**例子**

* `-rw-r--r--`：owner 可讀寫，其他人只能讀
* `drwxr-x---`：目錄 owner 可完全控制，group 可進入與讀，others 不可

**目錄的特別說明**

* 目錄 `r`：可列出內容
* 目錄 `w`：可增刪裡面的檔案
* 目錄 `x`：可進入該目錄

**現實應用**

* `/home/user` 常設成不給 others 存取
* `/var/www` 由網站服務帳號可讀、部署帳號可寫
* `/etc/shadow` 必須高度限制

---

## 4. ACL（Access Control List）

**定義**
ACL 是傳統 owner/group/others 之外的細粒度存取控制，可針對特定使用者或群組額外指定權限。投影片指出 ACL 是 fine-grained access control，且與傳統 permission model 共存。

**例子**

* 檔案 owner 是 Alice
* 但可以額外讓 Bob 有讀權限、讓 TA 群組有寫權限

**現實應用**

* 實驗課共享資料夾中，只給特定學生小組讀寫
* 研究室共用資料但只讓 PI 有刪除權限
* 公司專案資料夾分層授權

---

## 5. Authentication（身分驗證）

**定義**
驗證你是誰。投影片列出典型三類：

* something you know（密碼）
* something you have（卡片、token）
* something you are / do（指紋、生物特徵）

**例子**

* Linux login 密碼
* OTP token
* 指紋解鎖

**現實應用**

* VPN 登入
* GitHub 兩步驟驗證
* 公司 SSO + MFA

---

## 6. `/etc/passwd` 與 `/etc/shadow`

**`/etc/passwd` 定義**
儲存帳號基本資訊，不應再直接放密碼雜湊。投影片指出現代系統密碼不存這裡。

**`/etc/shadow` 定義**
儲存真正的密碼雜湊與密碼政策資訊，只允許高權限程式安全存取。投影片與 SetUID 投影片都直接以這個檔案說明權限困境。 

**為什麼重要**
如果一般使用者可讀到密碼雜湊，就可離線字典攻擊。

**現實應用**

* Linux 主機登入安全
* 伺服器帳號管理
* PAM、SSH 登入驗證

---

## 7. Salt（鹽值）

**定義**
在密碼雜湊時加入隨機值，使得相同明文密碼產生不同 hash，降低字典攻擊和彩虹表攻擊效果。投影片直接說 salt 的用途是對抗 brute-force / dictionary / rainbow table。

OS Concepts 10th 也指出 salt 會讓相同密碼產生不同 hash，並迫使攻擊者不能只預算一份字典 hash。

**例子**

* 兩個使用者都用 `password123`
* 有 salt 時，雜湊結果不同

**現實應用**

* Linux 本機帳號
* Web 平台密碼儲存
* IAM / identity systems

---

## 8. sudo

**定義**
以受控方式，讓授權使用者暫時以 root 或其他身分執行指令。投影片列為三種特權機制之一。

**例子**

* `sudo apt update`
* `sudo systemctl restart nginx`

**現實應用**

* 管理員維運
* 開發者部署
* 不直接開 root shell，降低失誤面積

---

## 9. POSIX Capabilities

**定義**
把 root 的「完整超級權限」拆成較小的權限片段。投影片說它把 root privilege 分成 smaller privilege units；OS Concepts 10th 進一步說 Linux capabilities 是把 root power 切成 bitmask，實作 least privilege。 

**例子**

* `CAP_NET_RAW`：允許建立 raw socket
* `CAP_NET_BIND_SERVICE`：允許綁定低於 1024 port
* `CAP_SYS_ADMIN`：非常強大，應避免濫用

**現實應用**

* `ping` 只需 `CAP_NET_RAW`，不必整個程式都 setuid root。投影片就用 ping 作為案例。
* Wireshark 使用 `dumpcap` 擷取封包，GUI 本身不需整體提權。投影片亦以此為案例。

---

# 四、Privileged Program 與 Set-UID

## 1. 為什麼需要 Privileged Programs？

Set-UID 投影片用最典型的例子：**一般使用者無法直接寫 `/etc/shadow`，那要怎麼改密碼？** 這就是 privileged program 存在的原因。

也就是說，OS 的一般存取控制故意把敏感資源鎖住，但某些業務又必須被允許，因此需要：

* 不是直接把使用者變 root
* 而是讓「特定程式」在「特定範圍」內代行特權

這正是講義應抓住的核心。

---

## 2. Set-UID（設定使用者識別位）

**定義**
Set-UID 是 UNIX/Linux 一種特殊檔案屬性。當可執行檔啟用 setuid bit 後，執行它的 process 會暫時以「檔案擁有者」的有效身分運作。投影片與 OS Concepts 10th 都有明確說明。 

**經典例子**

* `/usr/bin/passwd` 通常是 setuid-root
* 一般使用者執行 `passwd` 時，程式暫時具備足夠權限去改 `/etc/shadow`
* 但理想上只能改「自己的」密碼

---

## 3. RUID / EUID

Set-UID 投影片非常重要的一頁是這個。

### Real UID（RUID）

**定義**
真正啟動這個 process 的使用者身分。

### Effective UID（EUID）

**定義**
OS 在做 access control 時實際採用的身分。投影片強調 access control 是 based on EUID。

### 一般程式

* RUID = EUID = 使用者本人

### Set-UID 程式

* RUID = 呼叫者
* EUID = 檔案擁有者

**例子**

* 使用者 `alice` 執行 root 擁有的 setuid 程式
* RUID = alice
* EUID = root

**現實應用**

* `passwd`
* 某些掛載工具
* 某些 legacy 管理工具

---

## 4. Set-UID 與 sudo 的差別

### sudo

* 以「命令」為單位授權
* 有 sudoers policy
* 適合管理行為

### setuid

* 以「程式檔」為單位授權
* 執行即提權
* 適合很小、很單一、很受控的特權任務

### 直觀比較

* `sudo vi /etc/hosts`：你在 root 權限下開編輯器
* `passwd` setuid-root：只有這支程式在非常受限流程中操作敏感檔案

---

# 五、Set-UID 為什麼危險？

OS Concepts 10th 明說：setuid 程式的潛在威力很大，因此必須非常小心設計；歷史上大量 setuid binary 因 race condition 或 code injection 被利用，導致直接 root。

投影片則把攻擊面整理得更實務：

## 1. 使用者輸入攻擊

* Buffer overflow
* Format string vulnerability
* 沒有 sanitize 的輸入

**例子**
程式允許使用者輸入 shell path、檔名、帳號資料，但未正確檢查，導致竄改行為。

**現實應用**

* 老舊 UNIX 工具
* C 寫的特權工具
* 維運腳本包裝器

---

## 2. Environment Variable 攻擊

投影片直接點出：

* `PATH`
* `system()` 會先啟動 shell
* 若依賴環境變數找指令，可能被攻擊者控制

**危險範例**

```c
system("ls");
```

如果 PATH 被操控，執行到的可能不是你以為的 `/bin/ls`。

---

## 3. Command Injection

投影片有一句非常重要：
**Don’t mix code and data**。
並指出這正是 `system()`、XSS、SQL injection、buffer overflow 等共同問題。

**例子**

* 把使用者輸入直接拼到 shell command
* 特權程式把資料當成程式碼

**現實應用**

* Web backend 呼叫 shell
* 備份工具接受路徑參數
* setuid helper 執行外部命令

---

## 4. Race Condition / TOCTOU

投影片提到可經由 symbolic link、world-writable 目錄等方式影響特權程式，這正是典型 TOCTOU 問題。

**例子**

* 程式先檢查 `/tmp/target` 安全
* 下一秒攻擊者把它換成 symlink 指向敏感檔案
* 程式再用 root 權限去寫

**現實應用**

* `/tmp` 暫存檔
* lock file
* log rotation
* print spooler 類工具

---

## 5. Capability Leak（能力洩漏）

投影片用 file descriptor 沒關掉的例子說明：
程式先用 root 打開敏感檔，再降權，但忘了關 fd，之後啟動 shell 或其他程式，導致原本不該保留的能力被帶下去。

**核心觀念**
權限不只是 UID/GID，還包含已開啟的檔案描述元、socket、memory mapping 等「已取得能力」。

---

# 六、Set-UID / 特權程式的安全設計原則

這一段最值得放進講義，因為它把投影片、OS Concepts、Bishop 三者串起來。

## 原則 1：Least Privilege（最小權限原則）

OS Concepts 10th 對此的定義非常清楚：
每個程式與特權使用者，只應擁有完成工作所需的最少權限。

Bishop《Computer Security》也把它列為 secure design 第一原則，位於 Chapter 14, Section 14.2.1, p.457。

**實作建議**

* 能用 capability 就不要整個 root
* 能只在短時間提權就不要全程提權
* 能把敏感功能切出小 helper 就不要放進大程式

---

## 原則 2：Mechanism 與 Policy 分離

OS Concepts 10th 在 Protection 章強調：

* **mechanism**：怎麼做
* **policy**：允許做什麼
  分離兩者才有彈性。

**實作建議**

* 程式不要把所有授權邏輯寫死
* 權限判定與實際執行分開
* 對誰可用、做什麼，盡量可配置可審計

---

## 原則 3：不要混合 code 與 data

投影片把這條稱為 isolation principle。

**實作建議**

* 不用 `system()`
* 優先用 `execve()` 類明確分離 command 與 arguments 的介面
* 不拼 shell 字串
* 不相信 PATH、IFS、LD_* 等環境變數

---

## 原則 4：特權只保留到必要時刻

Bishop 在 Program Security 章的例子明確提到：

* 程式一開始可能需 root
* 但在打開 log file 與 access-control file 後，就應降回一般使用者權限
* 只有真正需要切角色時，再暫時使用更高權限
  這樣可以把高權限暴露時間降到最低。

**這一點非常適合當考點。**

---

## 原則 5：控制檔與設定檔也必須受保護

Bishop 也指出，不只是程式本身，連 access control file 都必須驗證完整性與權限，否則特權邏輯照樣被攻破。

**實作建議**

* 啟動時檢查 config owner / permission
* 檢查目錄權限
* 避免可被一般使用者改寫的控制檔

---

## 原則 6：優先用 capabilities 取代 full root

OS Concepts 10th 介紹 Linux capabilities 時清楚指出，它是對 UNIX 全有全無 root 模型的改良。

**現實例子**

* `ping` 用 `CAP_NET_RAW`
* `dumpcap` 用封包擷取所需能力
* container runtime 將 root 權限切片管理

---

# 七、重要專有名詞整理（可直接當講義 glossary）

## Privileged instruction

只能在 kernel mode 執行的指令，如 I/O control、interrupt management。

## Privilege escalation

本來不該有的更高權限被取得。OS Concepts glossary 直接如此定義。

## Protection

控制 process / user 對系統資源的存取。是「內部機制」問題。

## Security

防止未授權存取、惡意破壞、資料不一致與外部威脅。OS Concepts 把 security 與 protection 分開談。

## Setuid bit

啟用後，執行者暫時取得檔案擁有者的有效身分。

## Capability

被切分出來的特權單位，用 bitmask 控制。

## Compartmentalization

把系統分艙隔離；一個元件被攻破，不應連帶整個系統失守。OS Concepts 在 least privilege 後接著談到這點。

## Audit trail

可追查的操作紀錄。對偵測攻擊與事後鑑識非常重要。

---

# 八、現實生活中的應用案例

## 1. `passwd`

最典型 setuid-root 程式。
一般使用者不能直接改 `/etc/shadow`，但可透過受控流程改自己的密碼。

## 2. `ping`

不需要整個 root，只需要 raw socket 能力。
因此更適合用 capability 而非 setuid-root。

## 3. Wireshark / dumpcap

GUI 本身不該拿高權限；封包擷取功能獨立為特權較小的元件。
這是最小權限與功能切割的現代實例。

## 4. Web 伺服器

Nginx / Apache 常以較低權限帳號執行 worker，避免整個站台 process 都是 root。

## 5. Container / sandbox

把服務放入容器、namespace、sandbox，本質上就是 compartmentalization 與 least privilege 的延伸。OS Concepts 新版也把 application containment、sandboxing、system-call filtering 視為重要保護方法。

# 十二、可直接放在講義最後的總結

1. **Linux host security 的核心不是單純「限制」，而是「受控授權」**。
2. `/etc/shadow`、`passwd`、`sudo`、capabilities、ACL 都是在解同一個問題：
   **如何讓必要行為可被執行，同時又不讓整個系統被打開。**
3. **Set-UID 是歷史上非常重要的機制，但也是高風險設計。**
4. 現代安全設計更偏向：

   * least privilege
   * compartmentalization
   * capabilities
   * sandboxing
   * code/data separation
5. **Operating System Concepts 10th** 最適合拿來建立「作業系統如何原生支援安全」的觀念主線；
   **Computer Security** 最適合補設計哲學；
   **CEH** 最適合補攻擊視角；
   **Attacking Network Protocols** 最適合補漏洞與分析技術。
