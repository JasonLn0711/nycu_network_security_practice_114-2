# SetUID 與 Privileged Program 筆記

這份版本改成比較適合台灣高中生閱讀的白話說法。

目標不是只背：

「SetUID 就是一個會提權的 bit。」

而是真的知道：

- 為什麼作業系統會需要它
- 它到底怎麼運作
- 為什麼它理論上很有用
- 為什麼它實際上又超危險
- 程式設計上該怎麼避免踩雷

---

## 先講一句最重要的

**SetUID 可以把它想成：作業系統借給某個程式一張臨時特權通行證。**

一般使用者本來做不到的事，
如果這個程式被設計成「只能做某一種受控的高權限工作」，
那系統就允許它暫時用比較高的權限去做。

聽起來很合理，
因為有些事情確實不能直接把 root 交給每個人，
但又不能完全不讓人做。

可是問題也正是在這裡：

**只要這個有特權的程式寫得不好，它就不是工具，而是漏洞放大器。**

---

## 一、為什麼會需要 SetUID？

先不要急著背定義。
先想一個很現實的問題：

很多系統操作明明需要高權限，
但一般使用者又真的會有正當需求。

### 最經典的例子：改密碼

一般使用者不能隨便改 `/etc/shadow`，
因為這是非常敏感的系統檔案。

可是一般使用者又必須能改自己的密碼。

那怎麼辦？

答案不是：

「那就把 `/etc/shadow` 開放給所有人寫。」

這太危險了。

正確做法是：

**讓一個被嚴格控制的程式代替你完成這件事。**

像 `passwd` 這種程式，
就不是讓你直接碰敏感檔案，
而是讓你透過一個規則明確的工具來做。

### 白話理解

像學校的總務處有保險櫃，
學生不能直接拿總鑰匙。

但如果你真的有正當需求，
例如要補發學生證，
那學校可以設計一個固定流程：

- 先驗證你是誰
- 只能填特定表單
- 只能做特定動作
- 不能亂碰其他東西

SetUID 很像這種概念。

**不是把最高權限直接給你。**

而是給你一個「只能代辦某件高權限工作」的程式。

### 這裡真正要懂的事

SetUID 存在的原因不是因為 UNIX 很愛危險。
而是因為系統真的需要一種方式，
讓：

- 一般使用者
- 在不直接拿到 root 的前提下
- 完成少數必須高權限才能做的事情

---

## 二、SetUID 到底是什麼？

SetUID 是 executable file 上的一個特殊權限位元。

當一個程式檔被設成 SetUID，
而且有人去執行它時，
這個新產生的 process 在做權限檢查時，
會暫時拿到「程式擁有者」的身分。

### 先講最常見的情況

通常會長這樣：

- 檔案 owner 是 `root`
- SetUID bit 有開
- 一般使用者去執行它

結果就是：

**這個程式執行時的有效權限，會變成 root。**

### 在 `ls -l` 會看到什麼？

像這樣：

```bash
-rwsr-xr-x
```

你看到 owner 的 execute 那格變成 `s`，
那通常就是 SetUID 的表面痕跡。

如果你在教學示範裡看到像：

```bash
chmod 4755 some_program
```

那通常就是在把 SetUID bit 打開，
讓這個程式之後執行時可能帶有 owner 的有效權限。

### 白話理解

你可以把普通程式想成一般學生證。

SetUID 程式則像是某種「有額外授權的校內工作證」。

你本人還是學生，
但當你拿著那張工作證去做那件被允許的事情時，
系統會在某些門禁判斷上，
暫時讓你像工作人員一樣通過。

### 很重要的一句話

**SetUID 不是把使用者變成 root。**

它是讓某個 process 在執行期間，
暫時擁有比較高的有效權限。

這句一定要分清楚。

---

## 三、RUID 跟 EUID 是整章最核心的骨架

這裡如果搞不懂，
後面幾乎全部會亂掉。

Linux 中一個 process 常會牽涉到至少兩個身分：

### `RUID`：Real UID

代表真正啟動這個 process 的使用者是誰。

### `EUID`：Effective UID

代表核心在做 access control 時，
應該把這個 process 當成誰。

### 最重要的一句

**存取控制主要看的是 EUID，不是 RUID。**

### 普通程式的情況

如果你用一般帳號執行一般程式，
通常會是：

- `RUID = 你`
- `EUID = 你`

所以系統在判斷你能不能開檔案、能不能寫檔案時，
就是用你自己的身分。

### SetUID 程式的情況

如果這個程式檔案的 owner 是 root，
而且 SetUID bit 打開，
你去執行它時，
通常會變成：

- `RUID = 你`
- `EUID = root`

這就是為什麼它會有高權限效果。

### 白話理解

把 `RUID` 想成：

「你本人是誰」

把 `EUID` 想成：

「警衛現在要把你當成誰來決定能不能進門」

平常兩個一樣。

但 SetUID 的情況像是：

- 你本人還是學生
- 可是某個流程裡，警衛暫時把你當成有特別授權的人

### 這裡真的要背熟

**SetUID 的本質不是改變使用者本人。**

**而是改變 process 的有效身分。**

---

## 四、它到底怎麼運作？

你可以把流程想成這樣：

1. 你去執行一個程式檔
2. 核心發現這個檔案有 SetUID bit
3. 核心再看這個檔案 owner 是誰
4. 新 process 建立時，EUID 被設成那個 owner
5. 之後這個 process 在做權限檢查時，就以那個 EUID 為主

### 這裡有一個很容易忽略的點

很多人一看到 SetUID，
就只想到 root。

但其實不是只有 root 才能當 owner。

如果某個 SetUID 程式的 owner 是另一個普通使用者，
那它執行時拿到的就是那個人的有效權限，
不是 root。

### 白話理解

SetUID 不是一個「root 魔法開關」。

它比較像是：

**執行這個程式時，先切換成檔案主人的工作身分。**

如果檔案主人是主任，
那你就像暫時套上主任的權限外套。

如果檔案主人只是一般職員，
那你拿到的也只是職員等級的權限。

### 這段真正的技術重點

核心做的事其實很小，
但很有力量：

**在 `exec` 那一瞬間，根據檔案 metadata 改變 process 的 effective identity。**

之後能不能安全，
就高度依賴程式自己是不是寫得夠謹慎。

---

## 五、為什麼 SetUID 理論上是安全的？

它理論上安全的原因不是因為「提權很安全」，
而是因為它假設：

**提權只發生在一個被嚴格限制用途的程式裡。**

### 白話理解

像學校不會把總務處總鑰匙直接發給你。

但可以設計一台機器，
那台機器只能做一件事：

- 驗證你身份
- 確認你有資格
- 幫你開某一扇指定的門
- 開完就結束

如果它真的只能做這件事，
那風險就相對可控。

### 所以 SetUID 的安全前提是什麼？

一個 SetUID 程式理論上安全，
通常隱含這些條件：

1. 程式邏輯正確
2. 程式功能範圍很窄
3. 所有輸入都被正確檢查
4. 不會偷偷受其他隱藏因素影響
5. 提權只用在真正需要的那一小段

### 問題就在這裡

現實世界的程式很少真的完美。

所以只要以上任何一條失敗，
SetUID 就會從：

- 受控提權機制

變成：

- 提權漏洞的加速器

---

## 六、為什麼 SetUID 實際上很危險？

因為使用者能影響的東西，
往往比你以為的多很多。

很多人一想到輸入，
只想到：

- 鍵盤打字
- command line argument
- 表單欄位

但對一個 privileged program 來說，
會影響它的其實很多：

- 顯式 user input
- 檔案系統狀態
- 符號連結
- race condition
- environment variables
- 它呼叫的外部程式
- 它繼承到的 file descriptor
- 它 later 啟動的低權限 process

### 白話理解

這就像你讓一個學生穿著臨時主任通行證進主任室辦事。

你以為你只需要管他手上那張申請單。

但實際上你還要管：

- 他有沒有偷帶別張紙條
- 他旁邊的人有沒有偷偷換資料
- 他是不是從別的抽屜拿到不該拿的東西
- 他是不是去叫另一個人來幫忙做危險操作

所以 SetUID 的危險從來不只是：

「有沒有一個明顯的 buffer overflow。」

而是：

**整個高權限程式會不會被太多使用者可控制的因素牽著走。**

---

## 七、第一類風險：顯式使用者輸入

這是最直覺的一類。

如果 SetUID 程式直接吃使用者輸入，
而你又寫得不夠小心，
就可能出現：

- buffer overflow
- format string
- command injection
- 輸入驗證不完整

### 為什麼在 SetUID 程式裡特別嚴重？

因為這已經不是普通程式的小 bug。

一旦程式是用高權限在運作，
那任何輸入處理失誤都可能變成：

**低權限使用者藉由 bug 去操控高權限行為。**

### 真實世界感的例子

假設有個高權限程式負責幫你修改某個敏感系統設定，
它本來只應該接受：

- 合法帳號名稱
- 合法 shell 路徑
- 合法格式

但如果它沒有檢查好，
有人可能塞進超長字串、
奇怪字元、
甚至多餘欄位內容。

那它可能不只改設定，
還把整份系統資料寫壞。

### 白話理解

像學校櫃台本來只應該接受你填：

- 學號
- 姓名
- 申請原因

結果系統沒檢查好，
有人把整段亂碼和假指令都塞進去，
櫃台還照單全收。

那這就不是「資料錯了」而已，
而是整個高權限流程被你帶偏。

### 這段最重要的結論

**在 SetUID 程式裡，輸入驗證失敗不是普通 bug，而是 privilege boundary failure。**

---

## 八、第二類風險：系統輸入、檔案系統狀態、race condition

有些輸入不是從鍵盤來，
而是從系統狀態來。

例如：

- 某個檔案現在存在不存在
- 某個路徑現在指到哪裡
- 某個目錄是不是 world-writable
- 某個符號連結在你檢查後有沒有被換掉

### 什麼是 race condition？

簡單說就是：

**你檢查的時候看到的是 A，真正使用的時候已經變成 B。**

這種問題常被叫做 TOCTOU，
也就是：

- Time Of Check
- Time Of Use

### 白話理解

像你到櫃台前先看見抽屜裡是「白色申請書」。

你一轉身去蓋章，
旁邊的人立刻把抽屜裡的紙換成「別的文件」。

你回來時沒有重查，
就拿去蓋章了。

這就是 race condition 的味道。

### 符號連結為什麼危險？

假設高權限程式想安全地寫某個暫存檔，
它以為自己在寫普通檔案。

但如果攻擊者先在那裡放一個 symbolic link，
把它指到敏感檔案，
那高權限程式可能就會幫攻擊者去寫不該寫的地方。

### 真實世界類比

像主任要在某個資料夾放一份普通公告，
結果有人先在那個標籤位置放了一個假抽屜導引牌，
讓主任其實把資料放進成績櫃。

主任以為自己在做普通事，
實際上是在替攻擊者動敏感資料。

### 這段要記住的觀念

**在高權限情境裡，檔案系統狀態本身也可能是攻擊者準備好的輸入。**

所以不能只相信：

- 「我剛剛檢查過」
- 「這個路徑看起來沒問題」

因為問題常常發生在「檢查完之後、使用之前」。

---

## 九、第三類風險：Environment Variables

這一類你在 `note_2.md` 已經看過，
但放到 SetUID 情境下會更危險。

### 為什麼？

因為 environment variable 是使用者在程式啟動前就能設定的背景輸入。

如果高權限程式會受它影響，
那就等於：

**低權限使用者在程式開跑前，先偷偷決定了一部分高權限程式的行為。**

### 最常見例子：`PATH`

如果高權限程式沒有寫絕對路徑，
而是交給 shell 或其他機制靠 `PATH` 去找命令，
攻擊者就能改 `PATH`，
讓程式跑到錯的東西。

### 白話理解

像主任叫助理去拿：

「那個表單。」

但沒有說：

- 哪個櫃子
- 哪一層
- 哪一份才是真的

助理就照別人偷偷塞給他的搜尋順序去找。

那最後拿到的，
可能根本不是主任原本想要的表單。

### 不只 `PATH`

還有很多 environment variable 可能影響：

- linker 行為
- shell 行為
- locale / library 行為
- 程式自己用 `getenv()` 讀到的值

### 這段最關鍵的一句

**environment variable 不是看不見就代表安全，它只是藏得比較深的使用者輸入。**

---

## 十、第四類風險：非特權程序、外部程式、以及你叫出去的幫手

有時候高權限程式自己不直接做某件事，
而是去叫另一個程式幫忙。

這時風險不會消失，
反而常常更大。

### 為什麼？

因為你原本只有一個高權限程式要顧。

但一旦你又去呼叫：

- shell
- editor
- 外部工具
- 其他子程序

那整個 attack surface 就擴大了。

### 白話理解

像主任室裡本來只有主任和一套固定流程。

結果主任覺得麻煩，
又叫來：

- 助理
- 外包人員
- 自由出入的工讀生

然後說：

「你們幫我把事情做完。」

這樣出事機率當然暴增。

### 這類風險的核心觀念

**高權限程式一旦把事情交給其他程序做，權限邊界就會變得更難控制。**

尤其如果那個被叫來的程式本身又會：

- 看環境變數
- 解釋字串
- 讀設定檔
- 開互動介面

那風險就更大。

---

## 十一、Capability Leak 是什麼？這段超容易被低估

很多人以為：

「只要我最後把 EUID 降回來，就安全了。」

這句話不一定對。

因為權限不只存在於 UID，
也可能存在於你已經拿到手的資源。

### 最常見例子：file descriptor

假設一個 SetUID root 程式先做這些事：

1. 先以 root 權限打開某個只有 root 才能寫的檔案
2. 成功拿到 file descriptor
3. 之後把 EUID 降回一般使用者
4. 再啟動一個 shell 或繼續跑別的低權限流程

表面上看起來：

- 權限已經降了

但實際上：

- 那個已經打開的 file descriptor 還能用

這就叫 capability leak。

### 白話理解

像老師一開始帶你進器材室，
還把抽屜鑰匙交到你手上。

後來老師說：

「好，現在你不是管理員模式了。」

但鑰匙還在你手上。

那你其實還是能開那個抽屜。

### 為什麼這很重要？

因為這表示：

**降權不等於把之前拿到的能力全部一起清掉。**

### 工程上的真正重點

核心有些 access check 是在 `open()` 當下做的。

一旦檔案已經被高權限成功打開，
後面你就算把 EUID 降下來，
那個已經拿到的 descriptor 仍然是一種能力。

### 防禦方法

**在降權之前，先把不該留著的高權限資源關掉。**

尤其是：

- file descriptor
- socket
- 其他已經取得的 privileged handle

## 十二、`system()` 為什麼危險？

這是考試超愛考的地方。

你不能只背：

「`system()` 不安全。」

你要知道它到底危險在哪。

### `system()` 做了什麼？

`system()` 並不是直接幫你執行某個 binary。

它通常會先叫 shell，
再由 shell 去解釋你給它的那串命令字串。

這代表什麼？

這代表：

- shell interpretation 進來了
- environment variable 進來了
- `PATH` 進來了
- 萬用字元、重導向、分號、展開等語法也進來了

### 白話理解

像你不是直接把命令交給一台只會照做的機器。

你是先把一句話交給一個「會腦補、會解讀、會自己補規則」的人。

那當然比較危險。

### 最大問題一：資料和程式碼混在一起

如果你把一整串字串丟給 `system()`，
而這裡面又有一部分來自使用者輸入，
那資料很容易不小心變成命令語法的一部分。

這就是常見 command injection 的來源之一。

### 白話理解

像你本來只是要助理把「姓名」填進表單。

結果助理看到那欄文字時，
不只是把它當資料，
而是把它當成新的工作指令。

那就出事了。

### 最大問題二：shell 和 `PATH` 會介入

只要 shell 參與，
它就會去看很多背景資訊，
像是：

- `PATH`
- shell 展開規則
- 環境變數

這等於把本來就敏感的高權限流程，
再多接了一層會自動解讀字串的系統。

### 這段的真正結論

**在 SetUID 或任何 privileged program 裡，把高權限控制流交給 shell 幾乎總是危險的。**

---

## 十三、`execve()` 為什麼比較安全？

這裡也不能只背：

「`execve()` 比較安全。」

重點是：

**它比較安全，是因為它把 code 和 data 分得比較清楚。**

### `execve()` 的特性

你會直接指定：

- 要執行哪個 binary
- 參數陣列是什麼
- 環境要不要另外指定

這表示：

- command name 是程式自己決定的
- user input 只是 arguments
- 不需要先交給 shell 解釋

### 白話理解

像你不是跟助理說：

「你幫我處理一下這段話。」

而是直接寫成：

- 去這個地址
- 找這個人
- 交這份文件

這樣中間能被誤解的空間就小很多。

### 這段最重要的原則

**不要把 code 和 data 混在一起。**

這不只是 shell 的問題。

很多安全漏洞本質上都跟這件事有關，
例如：

- command injection
- SQL injection
- XSS

都是資料不小心變成程式碼的一部分。

### 但要小心

不是所有 `exec` 家族都一樣安全。

有些像：

- `execlp()`
- `execvp()`
- `execvpe()`

會去參考 `PATH`，
模仿 shell 找程式的方式。

所以在高權限情境裡，
這些也可能被 environment variable 影響。

### 真正比較安全的版本

在 SetUID 程式裡，
你要記的是：

**用 `execve()` + 絕對路徑 + 乾淨的參數與環境。**

---

## 十四、為什麼 `/bin/sh`、`vi` 這類程式不能隨便做成 SetUID？

這題很重要，
因為它在考你對「受控提權」的理解到底夠不夠。

### SetUID 程式應該長什麼樣？

它應該是：

- 功能單一
- 行為可預測
- 介面狹窄
- 容易審計

### 那 shell 或 editor 呢？

像 `/bin/sh`、`bash`、`vi`、`vim` 這種工具，
本質上是：

- 通用命令執行器
- 高互動工具
- 能跑很多你難以完全預測的操作

如果把這種東西直接做成 SetUID，
你其實就是在把一大包超強能力包上高權限外套。

這根本不是受控提權。

這是直接開洞。

### 白話理解

SetUID 適合的是：

「只能幫你去櫃台蓋一種章的機器。」

不適合的是：

「把整間主任室和全部抽屜鑰匙交給一個萬能助理。」

### 即使某些 shell 會自己 drop privilege 呢？

有些 shell 在偵測到自己從 SetUID context 啟動時，
可能會主動降權。

但你不能把這件事當成設計依賴。

因為那只是某個工具自己的補救措施，
不是你整個 privileged design 本身就安全。

---

## 十五、Least Privilege：最重要的設計原則

這是整章最後最關鍵的工程思維。

### Principle of Least Privilege 是什麼？

意思是：

**程式只應該拿到完成任務所需的最小權限。**

不需要的權限不要拿。
不再需要的權限要盡快放掉。

### 白話理解

像你進實驗室做一個簡單工作，
只需要一把抽屜鑰匙。

那你就不應該同時拿到：

- 總鑰匙
- 保險箱鑰匙
- 教師辦公室門卡

這些東西拿越多，
出事時破壞就越大。

### 放到 SetUID 程式裡，這原則怎麼落地？

1. 高權限程式碼區域要越小越好

能不提權就不提權。
必須提權的那段要越短越好。

2. 功能要單一

像 `passwd` 就應該專心做改密碼，
不要順便變成任意檔案操作工具。

3. 所有輸入都當成不可信

不只鍵盤輸入，
還包括：

- environment variables
- file system state
- symbolic links
- inherited file descriptors
- 子程序行為

4. 不要叫 shell

能不用 `system()` 就不要用。
改用 `execve()`。

5. 降權前先清理能力

尤其是 file descriptor。
不然只是看起來降權，
實際上還留著特權資源。

### Linux 上怎麼做降權？

常見會看到像：

- `seteuid()`
- `setuid()`

這些函式用來暫時或永久丟掉權限。

但再次強調：

**降 UID 不等於自動清光所有已取得的能力。**

---

## 十六、SetUID 跟 `sudo`、Capabilities、Service Approach 的關係

這一段是在幫你建立比較完整的系統觀。

### `sudo`

`sudo` 也是一種 privileged mechanism。

但它比較像：

**受政策控制的高權限代理人。**

你能跑什麼、
誰能跑、
需不需要密碼，
這些通常由設定檔決定。

### 白話理解

像你不是拿到永久通行證，
而是每次到櫃台申請：

- 哪個人
- 可以做哪件事
- 有沒有被允許

### POSIX Capabilities

這是比 SetUID 更細的權限模型。

它不一定要一次給整包 root。

而是把 root 的某些能力拆成比較小塊，
例如：

- 某種網路能力
- 某種檔案存取能力
- 某種系統管理能力

### 白話理解

這很像不是把整串總鑰匙給你，
而是只給你：

- 開某一扇門的鑰匙

這樣如果出事，
破壞範圍也比較小。

### Service Approach

另外一種更現代的設計思路是：

**不要讓高權限程式直接從使用者環境啟動。**

而是讓高權限服務自己在乾淨環境裡常駐，
使用者只能透過固定介面送請求。

### 白話理解

這就像你不能直接走進主任室操作系統。

你只能到櫃台填申請單，
由櫃台依照固定規則幫你辦。

這通常比 SetUID 更容易控管。

### 這段最終要懂的觀念

SetUID 很重要，
但不是唯一方法，
也不是永遠最好的方法。

如果你要：

- 更細的權限控制
- 更小的 attack surface
- 更容易審計的設計

那 capabilities 或 service-based design 往往更合理。

---

## 十七、把整章壓成考試能寫的理解模型

你可以把這章整理成下面這條邏輯線：

1. 某些操作需要高權限，但不能直接把 root 給一般使用者。
2. 所以作業系統提供 SetUID，讓受控程式能暫時以檔案 owner 的權限執行。
3. 核心做 access control 時主要看 EUID，不看 RUID。
4. 一般程式 RUID = EUID；SetUID 程式則 RUID 仍是呼叫者，EUID 變成檔案 owner。
5. 這樣可以讓一般使用者透過特定程式完成必要的高權限操作，例如改密碼。
6. 但一旦程式寫不好，任何輸入問題都可能變成 privilege escalation。
7. 攻擊面不只 user input，還包括：
   - system state
   - race condition
   - environment variables
   - 外部程式與 shell
   - leaked capability
8. `system()` 危險在於 shell 介入、code/data 混合、`PATH` 和環境變數介入。
9. `execve()` 較安全在於 command 與 data 分離，但仍要搭配絕對路徑和乾淨環境。
10. 正確設計原則是 least privilege、縮小高權限區域、驗證所有輸入、避免 shell、降權前清理已取得的能力。

---

## 十八、最短背誦版

**SetUID 是一種受限提權機制，讓程式在執行時暫時以檔案 owner 的權限運作。**

**SetUID 不是把使用者變成 root，而是把 process 的 EUID 切成較高權限的身分。**

**核心做 access control 時主要看 EUID，所以 SetUID 程式可以代表使用者完成特定高權限工作，例如修改密碼。**

**它危險的原因是 privileged program 會受到 user input、system state、environment variables、外部程式與 leaked capability 的影響。**

**`system()` 危險在於 shell interpretation 和 code/data 混合；`execve()` 較安全在於 command 與 data 分離。**

**真正安全的設計原則是 least privilege、最小高權限程式碼、絕對路徑、避免 shell、清理 capability、並在能選時用更細緻的權限模型。**

---

## 十九、如果你想用一句超白話記住這章

**SetUID 就像系統借給某個程式一張臨時高權限工作證。**

**如果那個程式只做單一、可控、驗證完整的工作，這張工作證很好用。**

**如果那個程式會亂吃輸入、亂信環境、亂叫 shell、亂留資源，那這張工作證就會變成攻擊者的捷徑。**
